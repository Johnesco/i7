Sound Overlay

How ambient audio works in v3

Version 3 of Zork I adds ambient background audio — forest sounds outdoors, dripping water underground, machinery at the dam. The audio system is implemented entirely outside the game as a JavaScript overlay that watches the interpreter's status bar for room changes and crossfades between looping audio files based on which zone the player is in.

This approach works because all three web interpreters (Quixe, Parchment, Glulxe) render the status bar as HTML elements in the DOM. The overlay doesn't modify the game binary at all — it's a separate script loaded alongside the interpreter.

Architecture

The entire system lives in a single file: v3/lib/ambient-audio.js (367 lines). It runs as an IIFE that sets up a MutationObserver on the interpreter's status bar element, detects room changes, and manages audio playback with smooth crossfades.

How It Works

1. Status Bar Observation

When the page loads, the script polls for a .GridWindow element (the status bar container used by GlkOte-based interpreters). Once found, it attaches a MutationObserver that fires whenever the status bar content changes: 

var observer = new MutationObserver(function () {
  var name = extractRoomName(grid);
  if (name) onRoomChange(name);
});

observer.observe(grid, {
  childList: true,
  characterData: true,
  subtree: true,
});

2. Room Name Extraction

The status bar displays the room name followed by score and move count, separated by whitespace. The extractRoomName function reads the first .GridLine, trims it, and strips everything after two or more consecutive spaces: 

// Status line: "  West of House          Score: 0  Moves: 1"
text = text.trim();
text = text.replace(/\s{2,}.*$/, '');  // strip score/moves

3. Zone Lookup

The extracted room name is looked up in ROOM_ZONES, a flat object mapping ~70 room names to one of 10 zones. Rooms not in the map default to silence (no audio). If the zone hasn't changed since the last room transition, no audio action is taken.

4. Crossfade

When the zone changes, the old audio fades out and the new audio fades in simultaneously over 1000ms. When there is no previous audio playing (e.g. on the first room after page load), the fade-in is slower — 3000ms — for a gentle introduction. Fading uses requestAnimationFrame for smooth linear interpolation. Each zone has a preset target volume (ranging from 0.02 to 0.07) to balance the different audio loops. 

function fadeAudio(el, targetVol, doneCallback) {
  var startVol = el.volume;
  var startTime = performance.now();

  function step(now) {
    var elapsed = now - startTime;
    var t = Math.min(elapsed / FADE_MS, 1);      // 0..1
    el.volume = startVol + (targetVol - startVol) * t;
    if (t < 1) requestAnimationFrame(step);
    else if (doneCallback) doneCallback();
  }
  requestAnimationFrame(step);
}

When the old audio finishes fading out, it is paused and its source is cleared to release browser resources. The new audio element is created fresh each time via document.createElement('audio').

5. Autoplay Handling

Browsers block audio autoplay until the user interacts with the page. The overlay handles this by catching the rejected play() promise and registering click/keydown listeners that retry playback on the first user interaction. Since the player types commands to play the game, audio typically starts within seconds of the first room change.

6. Mute Button

A small speaker icon is rendered as a fixed overlay in the bottom-right corner. Clicking it toggles mute, fading the current audio to zero or back to its zone volume. The mute state is persisted in localStorage (key: zork1-audio-muted) so it survives page reloads.

River disambiguation

The Frigid River spans five rooms (River 1–5), but the status bar displays the same name — “Frigid River” — for all of them. Rivers 1–3 are calm; Rivers 4–5 are rapids approaching the falls. Since the overlay can only read the room name, it maps all Frigid River rooms to the calm “water” zone. Players near the falls hear rapids audio from the Aragain Falls and Canyon Bottom entries instead.

Engine Compatibility

The overlay works identically with all three interpreters because it operates entirely outside the game. It only reads the DOM — specifically the .GridWindow .GridLine elements that all GlkOte-based renderers produce for the status bar. The game binary is unmodified; the script is loaded via a <script> tag in each player page.

Engine Player Page How Audio Is Loaded
Quixe v3/index.html <script src="lib/ambient-audio.js"> + <script src="lib/sound-effects.js">
Parchment v3/parchment.html <script src="lib/ambient-audio.js"> + <script src="lib/sound-effects.js">
Glulxe v3/glulxe.html <script src="lib/ambient-audio.js"> + <script src="lib/sound-effects.js">

Since all three engines use GlkOte for their display layer, they all render the status bar with the same DOM structure. The observer attaches to whichever .GridWindow element appears first, regardless of which engine created it.

Limitations

  • Room changes only — The overlay can only detect when the status bar updates with a new room name. It cannot detect in-game events like combat, item pickups, or puzzle completions.
  • Text-matched sound effects — One-shot sound effects (16 triggers) are fired by pattern-matching game text output. This works for predictable text but cannot detect events that don't produce unique output strings.
  • No in-game control — The player cannot control audio from within the game (e.g., “turn off music”). The only control is the external mute button.
  • Ambiguous room names — Some rooms share status bar names (all Frigid River rooms show “Frigid River”, all maze rooms show “Maze”). The overlay cannot distinguish between them and maps them all to the same zone.
  • Autoplay restrictions — Browsers require a user interaction before audio can play. The first room change after page load may be silent until the player types a command.

Sound Effects

In addition to looping ambient audio, v3 includes one-shot sound effects triggered by specific in-game text. These are handled by a separate script: v3/lib/sound-effects.js.

How It Works

A second MutationObserver watches the .BufferWindow (the main game text output area) for newly added DOM nodes. When a node's text content matches a trigger pattern, the corresponding audio clip is played once at a preset volume. The clip's <audio> element is automatically removed when playback finishes.

Cooldowns

Each trigger has a cooldown period to prevent the same sound from firing repeatedly. The songbird text appears every turn in forest rooms (via a recurring game rule), so it uses a 10-second cooldown. The sword glow and thief laugh use 30-second cooldowns since they can fire repeatedly during gameplay. All other triggers use 5-second cooldowns as a guard against rapid re-triggering.

Mute Sync

Sound effects share mute state with ambient audio via the same localStorage key (zork1-audio-muted). When the player clicks the mute button created by ambient-audio.js, sound effects are also silenced. No duplicate mute button is created.

Adding New Triggers

To add a new sound effect, add an entry to the SOUND_TRIGGERS array in sound-effects.js: 

{ id: 'unique-name',
  pattern: /text pattern to match/i,
  src: 'audio/sfx/filename.mp3',
  volume: 0.3,
  cooldownMs: 5000 }

Place the MP3 file in v3/audio/sfx/. The pattern is tested against each newly added DOM node's textContent, so it should match a substring of the game output that uniquely identifies the event.

Future Direction

The overlay approach was chosen for v3 as a quick, non-invasive way to add atmosphere without modifying the game source. It works well for ambient loops but cannot support event-driven sound effects or in-game audio control.

The path forward is native Glk sound (#60) — using the Glulx virtual machine's built-in sound channel support to play audio from within the game itself. This would allow:

  • Sound effects triggered by game events (combat hits, doors, puzzles)
  • In-game volume control and mute commands
  • Accurate per-room audio without status bar parsing
  • Multiple simultaneous sound channels (ambient + effects)

Native Glk sound requires the Glulxe (WASM) engine, which is currently the only web interpreter with Glk sound channel support. Quixe and Parchment do not implement glk_schannel_* functions. A future version could use Glulxe-native sound as the primary system with the JS overlay as a fallback for other engines.
